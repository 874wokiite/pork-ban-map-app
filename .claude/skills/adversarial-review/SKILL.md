# 敵対的レビュー Skill

## 目的

「このコードは正しいか?」ではなく **「このコードをどうやったら壊せるか?」** を問うレビュー。
実装者の意図を信用せず、検察官として振る舞い、具体的な失敗シナリオを突きつける。

## 大原則

1. **すべての指摘に「壊し方」を添える。** 「〜が心配」「〜のほうが良い」は指摘ではない。「この入力・この状態のとき、この行でこう壊れる」まで言えて初めて指摘。構築できなければ捨てる。
2. **指摘する前に必ず検証する。** 疑わしい箇所を見つけたら、実際のコードパスを読み切り、可能ならBashで再現(型チェック、テスト、Nodeでのスニペット実行)してから報告する。未検証の推測を報告しない。
3. **原則修正しない。ただし Critical は例外。** 通常の指摘はレビュー結果の報告のみで、修正はユーザーの指示を待つ。ただし「これを直さないと絶対にリリースできない」レベル(= Critical かつ CONFIRMED)の指摘は、レビュー報告を出した後、**ユーザーに確認せず自動で修正に着手する**(手順6参照)。
4. **褒めない。** 良い点の列挙は不要。問題がなければ「N観点で攻撃したが壊せなかった」とだけ報告する。

## 手順

### 1. スコープ確定

- 対象が指定されていなければ `git diff HEAD`(未コミット変更)→ なければ `git diff main...HEAD`(ブランチ差分)の順で対象を決める。
- 差分だけでなく、**変更されたコードの呼び出し元・呼び出し先**まで読む。バグの多くは差分の外(呼び出し規約の破壊)にある。

### 2. 意図の把握

- 関連するspec・タスクファイル・コミットメッセージ・会話履歴から「この実装が何をするはずだったか」を確定する。
- 仕様が曖昧な部分は「実装者が勝手に決めた仕様」としてマークし、攻撃対象にする。

### 3. 攻撃(以下の観点を順に適用)

各観点で「壊せるか?」を自問し、壊せそうなら具体的な入力・状態・操作手順を構築する。

#### A. 境界と異常入力
- 空(空文字、空配列、空オブジェクト、0件データ)、null / undefined、境界値(0, -1, 最大長, off-by-one)
- 不正な形式(想定外の型、壊れたJSON、欠損フィールド)、Unicode(絵文字、結合文字、全角半角)
- 巨大入力(1万件の配列、長大文字列)でのパフォーマンス崩壊

#### B. 状態と時間
- 非同期の順序逆転(遅いリクエストが後から返って新しい状態を上書きする)
- 競合状態(連打、二重送信、同時実行)
- stale state / stale closure(Reactなら古いstateを掴んだままのコールバック)
- 初期化前アクセス、アンマウント後のsetState、クリーンアップ漏れ

#### C. エラーパス
- 例外・rejectが握りつぶされていないか(catchして無視、エラー時にUIが固まる)
- 失敗時の中途半端な状態(半分だけ書き込まれたデータ、ロールバックなし)
- ネットワーク断・タイムアウト・APIの5xx/4xxで何が起きるか

#### D. 仕様との乖離・暗黙の仕様変更
- 要求されたことを本当に全部やっているか(サボり、TODOの放置、ハッピーパスのみの実装)
- 要求されていないことを勝手にやっていないか(既存挙動のサイレント変更、勝手なフォールバック)
- 既存の呼び出し元が壊れていないか(シグネチャ変更、戻り値の形の変更、削除されたexport)

#### E. データ整合性
- スキーマとの不一致(JSONの形、必須フィールド、enum値のtypo)
- 永続データの後方互換(既存データに新コードを当てたとき壊れないか)
- 重複・一意性・参照整合性(IDの衝突、孤児レコード)

#### F. セキュリティ
- 入力のサニタイズ漏れ(XSS、インジェクション、パストラバーサル)
- 秘密情報の露出(ハードコードされたキー、クライアントに漏れる環境変数、ログへの出力)
- 認可の欠落(誰でも叩けるAPI、クライアント側だけのチェック)

#### G. 型とテストの欺瞞
- `any` / `as` キャスト / non-null assertion (`!`) が隠している嘘
- 通るけど何も検証していないテスト(アサーションなし、モックが本体をバイパス、実装をコピーしただけの期待値)
- 型は通るが実行時に壊れる箇所(JSONパース結果の無検証キャストなど)

#### H. UI規約準拠(UI変更を含む場合のみ)
`.claude/skills/ui-implementation/SKILL.md` を読み、変更されたUIコードをその規約と突き合わせる:
- カラーのハードコード(`#F32D00` や `text-red-500` の直書き)— カスタムカラー変数(`bg-primary`、`bg-accent-green` 等)を使うべき箇所
- ダークモード対応漏れ(`dark:` プレフィックスがない背景・文字・ボーダー)
- フォント指定漏れ(モーダル・カードに `var(--font-zen-maru-gothic)` がない)
- レスポンシブ対応漏れ(`md:` なしの固定レイアウト)、ホバーの `transition-colors` 漏れ
- 既存コンポーネントパターン(モーダル、ボタン、タグ、豚鼻スライダー等)からの逸脱、`aria-label` 等のアクセシビリティ欠落

この観点の「壊し方」は「ダークモードで背景が白のまま文字も白になり読めない」のように、**規約違反が引き起こす実際の見た目・体験の破綻**として書く。単なる「規約と違う」だけならMedium止まり。

### 4. 検証パス

見つけた候補それぞれについて:
1. 該当コードパスを最初から最後まで読み、本当にそのシナリオで壊れるかを確認する
2. 可能なら再現する(`npx tsc --noEmit`、既存テスト実行、scratchpadでの最小再現スクリプト)
3. 「実は防がれていた」ものは黙って捨てる。確信が持てないものは CONFIRMED / PLAUSIBLE を明示して区別する

### 5. 報告

深刻度順に報告する:

```
## 敵対的レビュー結果

### 🔴 Critical — データ破壊・セキュリティ・クラッシュ
1. **[要約]** `path/to/file.ts:42` (CONFIRMED)
   - 壊し方: [具体的な入力・状態・操作手順]
   - 結果: [何がどう壊れるか]
   - 修正方針: [1行で]

### 🟠 High — 特定条件で誤動作・仕様違反
### 🟡 Medium — エッジケースでの劣化・将来の地雷
### ⚪ 攻撃したが壊せなかった観点
- [観点名]: [何を試したか1行]
```

- 指摘ゼロの場合も「壊せなかった観点」一覧は必ず出す(何を試した上での「問題なし」かが価値)
- 好みの問題(命名、スタイル)は深刻度に関係なく報告しない

### 6. Critical の自動修正

レビュー報告を出した後、**Critical かつ CONFIRMED** の指摘があれば、ユーザーの指示を待たずにそのまま修正に着手する。

- 対象は「これを直さないと絶対にリリースできないもの」のみ: データ破壊、セキュリティ欠陥、クラッシュ、主要フローの機能停止。High以下は自動修正しない(修正方針の提示まで)
- 報告の末尾に「Critical N件を自動修正します」と宣言してから始める
- 修正は指摘した失敗シナリオを潰す**最小限の変更**に留める。ついでのリファクタリングをしない
- 修正後、手順4と同じ方法(該当パスの読み直し、`npx tsc --noEmit`、テスト、再現スクリプト)で**元の壊し方が通用しなくなったこと**を再検証する
- 最後に「何をどう直したか」「再検証の結果」を報告する。修正が大掛かりになりそう(複数ファイルの設計変更が必要など)な場合は自動修正せず、その旨を報告してユーザーの判断を仰ぐ
